# ADR 012 - Sin sistema de caché en el MVP

**Fecha**: 2025-10-14  
**Estado**: Aceptado (temporal para MVP)  
**Autor**: A. Administrador

## Contexto y Problema a Resolver

El backend realizará consultas a la base de datos PostgresSQL para cada búsqueda. Con ~1,000 usuarios, podrían generarse muchas consultas idénticas en periodos cortos.

**Alternativas consideradas**:

- Sin caché  
- Caché en memoria (Node.js)  
- Redis  
- Caché HTTP del navegador  

Implementar caché añade complejidad en invalidación, infraestructura, código y debugging.

## Decisión Tomada

Se decide **no implementar un sistema de caché en el MVP** de Curse Fast, priorizando simplicidad y velocidad de desarrollo.

## Consecuencias

### Positivas

- Código más simple y mantenible  
- Datos siempre actualizados  
- Sin problemas de invalidación  
- Desarrollo más rápido  
- Sin infraestructura adicional  
- Debugging simplificado  

### Negativas o Riesgos

- Mayor carga en la base de datos  
- Latencia incrementada en búsquedas  
- Costos potenciales mayores en bases de datos gestionadas  
- Escalabilidad limitada sin caché  
- Experiencia de usuario subóptima en picos de tráfico  

**Estrategia de Mitigación**:

1. Monitorear métricas de rendimiento desde el lanzamiento  
2. Implementar caché si el percentil 95 supera 300ms o hay sobrecarga  
3. Optimizar índices en MySQL como primera medida  
4. Usar caché HTTP del navegador para recursos estáticos del frontend
